# Picstash 作業の進め方

## 基本方針

- **対話的に進める**: 各ステップでユーザーと確認しながら実装
- **早めのフィードバック**: 動くものを早く見せて方向修正
- **小さく作って確認**: 一度に大きく作らず、小さな単位で確認

## タスクの流れ

### 1. タスク開始前

1. `.ai-agent/tasks/YYYYMMDD-タスク名/README.md` を作成
2. 以下の内容を記載:
   - 目的・ゴール
   - 実装方針
   - 完了条件
   - 作業ログ
3. ユーザーに方針を確認してから実装開始

### 2. 実装中

1. タスクリスト（TaskCreate/TaskUpdate）でタスクを細分化して進捗管理
2. 各ステップで動作確認
3. 必要に応じてユーザーに確認・フィードバックをもらう
4. **コード変更後は `npm run lint` を実行して通すこと**

### 3. タスク完了時

1. 完了条件をチェック
2. 作業ログに結果を記載
3. ユーザーに完了報告
4. 次のタスクに進むか確認

## 確認タイミング

- データモデル設計後
- API 実装後（curl などで動作確認）
- UI の骨組み完成後
- 各機能の実装完了後

## ブラウザでの動作確認

- **Playwright MCP を使用**: ブラウザでの動作確認には Playwright MCP を活用
- スナップショット（`browser_snapshot`）で画面状態を確認
- クリック・入力などの操作を自動化してテスト
- コンソールエラーやネットワークリクエストの確認も可能

## ドキュメント構成

```
.ai-agent/
├── steering/
│   ├── product.md    # プロダクト概要
│   ├── tech.md       # 技術概要
│   ├── plan.md       # 実装計画（セグメント一覧）
│   ├── market.md     # 市場分析
│   └── work.md       # 作業の進め方（本ファイル）
├── projects/
│   └── YYYYMMDD-プロジェクト名/
│       └── README.md # プロジェクト詳細（目標・タスク分解・進捗）
├── tasks/
│   └── YYYYMMDD-タスク名/
│       └── README.md # タスク詳細
└── structure.md      # ディレクトリ構成
```

## プロジェクトとタスク

### 使い分け

| | タスク | プロジェクト |
|---|---|---|
| スコープ | 単一機能・修正 | 複数タスクからなる大きな目標 |
| 期間 | 1日〜数日 | 数週間〜数ヶ月 |
| 例 | 「画像詳細表示を実装」 | 「Phase 1: デスクトップアプリ化」 |
| コマンド | `/autodev:start-new-task` | `/autodev:start-new-project` |

### プロジェクトの進め方

1. **計画**: プロジェクト README でタスク分解と依存関係を整理
2. **実行**: 各タスクを `/autodev:start-new-task` で個別に実行
3. **追跡**: プロジェクト README で進捗を管理

### タスクの依存関係

プロジェクト内のタスクには依存関係がある場合がある:
- タスク開始前に、依存タスクが完了しているか確認
- 依存関係図で全体の流れを把握
- 並行して進められるタスクを特定
